# Portfolio Framework And Visual Archive Design

Status: Approved direction, awaiting written-spec review

Date: 2026-07-15

This document amends, but does not replace,
`2026-07-14-yangjing-portfolio-design.md`. The original identity, Interface
X-Ray visual system, bilingual contract, evidence rules, privacy rules, and
publication gate remain authoritative unless this amendment explicitly changes
them.

## 1. Purpose

Build the complete visual and navigational framework before every final asset is
available. The local portfolio should already communicate Yang Jing's hierarchy,
range, and interaction quality while keeping every incomplete claim visibly and
technically separated from publishable evidence.

The framework must:

- treat visual impact and visual-expression ability as the portfolio's primary
  experience goal, with case-study reading efficiency as the secondary goal;
- use a wave-shaped visual rhythm: the Hero is the strongest peak, project
  transitions provide distinct secondary peaks, and evidence-reading surfaces
  become deliberately quiet and stable;
- establish the dual-identity Hero and editorial homepage rhythm;
- provide stable draft routes for ByteDance and Meeting;
- preserve the completed Call Agent and STT Demo experiences;
- add AIDX as a core live website that opens externally;
- reserve a lightweight Visual Archive for seven or eight image-led projects;
- avoid invented screenshots, metrics, client claims, contacts, or biographical
  facts;
- remain blocked from production publication until all required inputs are real.

## 4. Hero Draft State

The Hero retains the approved Adham Dannaway-inspired dual-role concept while
remaining original in execution.

The first viewport includes:

- `Yang Jing` as the primary heading;
- `Product Designer` and `AI-native Builder` as equal role signals;
- supporting copy covering large-scale consumer products, AI/B2B systems,
  Vibe Coding, prototyping, and shipping;
- a central stable portrait frame;
- the top edge of the ByteDance band.

Until the real portrait is supplied, the central frame uses a deliberate draft
media treatment: Interface X-Ray grid lines, `YJ` initials, stable dimensions,
and a compact Draft status. It must not resemble a generated person, silhouette,
or fabricated photograph. Production validation continues to require
`public/images/profile/yang-jing-hero.avif`.

The dual reveal uses direct manipulation as its primary interaction. On desktop,
the visitor drags the central divider to control the visible balance between the
two aligned portrait treatments. Hover does not move the divider. Hovering the
Builder field instead activates an abstract code atmosphere made from characters,
coordinates, data traces, and layered pointer parallax. The effect prioritizes
visual impact over source-code readability and remains subordinate to the
portrait and role labels.

At rest, the Builder atmosphere remains faintly active with low density and slow
drift. Hovering the Builder field or dragging the divider toward it increases
the field's density, velocity, clarity, and depth. Touch devices use divider
position as the intensity input so the atmosphere remains complete without
hover.

The code atmosphere is strictly clipped to the Builder layer. As tokens approach
the divider, they compress, shear, and redirect along its edge. This makes the
divider feel like a physical boundary while keeping the Designer field visually
clean.

Clicking either role animates the divider toward that identity. Keyboard arrow
keys provide an equivalent divider control. Mobile supports touch dragging and
retains the `DESIGN / DUAL / BUILD` segmented control. The divider holds its
position after release, with subtle magnetic settling at the center and near-edge
reveal states.

The divider does not reset while the visitor remains in the Hero. Five seconds
after the pointer leaves the Hero, it returns to center with damped motion.
Pointer re-entry, dragging, clicking, keyboard input, or touch immediately
cancels the reset. The timer does not animate while the document is hidden.
Reduced-motion mode removes perpetual drift, parallax, and timed auto-return
while preserving drag, click, touch, and keyboard control.

## 5. Core Project Treatments

### ByteDance

ByteDance is the first and largest project band. It uses a wide editorial layout,
an oversized `01`, consumer-scale language, verified role/status fields when

The local Draft route contains only approved framing:

- project identity and evidence boundary;
- a short public-safe proposition;
- role and status only when supplied or already verified;
- a PDF area that visibly remains unavailable until the approved file exists;
- the canonical previous/next project navigation.

It must not reconstruct the ByteDance case or infer content from private work.

### Call Agent

Call Agent uses its existing real media and native bilingual case. Its homepage
band should demonstrate the strongest available product screenshot and link to
the current internal route. No claim, status, evidence label, checksum, or PDF
behavior changes in this framework phase.

### Meeting

Meeting uses a shorter coral-accented interaction band. The framework shows the
shape of a real-time state sequence through three stable stages without drawing
fake product UI.

The local Draft route is structured around:

- making dynamic real-time collaboration visible and controllable;
- shipped evidence;
- interaction states and control complexity;
- a clearly labeled 2026 retrospective layer;
- limitations and evidence boundaries.

Reserved media positions correspond only to the required Meeting files already
defined by the publication manifest. No outcome metrics or customer audience
claims may be added.

### AIDX

AIDX is a compact but high-fidelity `Live Website` band placed after Meeting and
before STT Demo. It uses a real screenshot captured from the public website and
the verified scope:

- UI/UX Design;
- Information Architecture;
- Motion.

The band includes `Live Website` status and a `Visit live site` command with an
external-link icon. The main visual and command open
`https://aidxtech.com/` in a new tab with safe external-link attributes and an
accessible external-destination label.

AIDX has no internal detail route, no invented design analysis, no performance
metrics, and no business-result claim. A dated capture caption distinguishes the
portfolio screenshot from a permanent record of the site.

### STT Demo

STT Demo remains the sole Build Lab project. Its band stays dark, focused, and
more technical than the case-study bands. It links to the existing internal
Build Lab route and the verified local demo. The fixed source revision and

## 6. Visual Archive

The Visual Archive is an editorial image wall, not a repeated marketing-card
grid. It supports seven or eight entries with mixed but controlled aspect ratios.

Each real archive entry contains only:

- a unique stable key;
- localized project name;
- one approved image with intrinsic dimensions and meaningful alt text;
- a short category;
- Yang Jing's verified role;
- optional verified year;
- optional public external URL.

Default interaction opens the image in the existing accessible Lightbox. An
entry with a real public URL also exposes a separate external-link command. No
archive entry receives an internal detail route.

Desktop uses a 12-column editorial composition with wide, square, and portrait
frames arranged in a controlled sequence. Mobile becomes a single-column image
feed with the title, category, and role permanently visible below each image.
Hover-only information is forbidden.

Before real archive data is supplied, local development may render seven neutral
numbered media slots to validate composition. Draft slots contain no client
name, project claim, screenshot, external URL, or fake metadata. They are not
valid publication content and remain excluded from production output by the
publication gate.

## 8. Draft Semantics And Publication Boundary

Draft content is a development state, not public evidence.

- Draft routes use a visible localized `Draft` status.
- Draft media frames use stable dimensions so final assets do not reflow pages.
- Draft status is not encoded only by color.
- Missing assets never fall back to remote stock photography or generated UI.
- Draft routes remain in the local framework for navigation and responsive QA.
- Source and output publication validation explicitly reject remaining Draft
  markers; publication must not rely only on a coincidentally missing file.
- `npm run build` and `npm run verify:publish` continue to fail until every
  required real input exists and Draft publication requirements are satisfied.
- Cloudflare deployment must continue to use the guarded `npm run build`.

The development validator may report missing inputs without failing. Source and
output validation remain non-zero for incomplete publication content.

## 9. Component And Content Boundaries

The framework should add focused modules rather than placing homepage behavior
inside the route file:

- `DualIdentityHero`: server-rendered identity and draft/real media states.
- `FeaturedWork`: owns the approved homepage order.
- `FeaturedProject`: shared editorial semantics without enforcing one visual
  layout on every project.
- `MeetingPreview`: three-stage interaction composition.
- `LiveWebsiteProject`: external-only AIDX treatment.
- `BuildLabPreview`: the single STT Demo treatment.
- `VisualArchive`: validated real entries or explicit development slots.
- `DraftCase`: honest ByteDance and Meeting route scaffolding.

Project type, evidence depth, destination type, and availability are explicit
data fields. The UI must not infer them from missing URLs or images.

Suggested destination values are:

- `internal-case`;
- `external-live-site`;
- `lightbox-only`.

Suggested availability values are:

- `complete`;
- `draft`;
- `awaiting-assets`.

AIDX uses `external-live-site`. Visual Archive uses `lightbox-only` with an
optional secondary external URL. ByteDance and Meeting use `internal-case` with
Draft availability until approved assets exist.

## 10. Responsive And Accessibility Requirements

- At `1440x900` and `390x844`, both Hero identities and a hint of ByteDance are
  visible without incoherent overlap.
- Every fixed-format media frame uses explicit dimensions or aspect ratio.
- Every command target is at least `44x44px`.
- External links name their destination behavior and use a familiar icon.
- Draft status is announced as text.
- Archive information is visible without hover on touch devices.
- Lightbox Escape, focus trap, focus return, and body-scroll restoration remain
  mandatory.
- Reduced motion removes scan offsets, entrance transforms, and nonessential
  state transitions.
- No section introduces nested cards, excessive rounding, horizontal overflow,
  or viewport-width font scaling.

## 11. Verification

Implementation is accepted only when:

- the bilingual homepage renders the approved five-project sequence;
- AIDX opens only the verified external website and has no internal route;
- exactly one Build Lab entry exists;
- ByteDance and Meeting Draft routes are keyboard-accessible and clearly marked;
- Call Agent and STT regression contracts remain unchanged;
- seven or eight archive slots can be composed without changing layout code;
- real archive entries require intrinsic media dimensions and alt text;
- desktop and mobile screenshots show no overlap, blank accidental media,
  horizontal overflow, or hidden essential information;
- reduced-motion output is static and readable;
- framework/unit/browser tests pass;
- publication validation still blocks the incomplete site.

## 12. Deferred Inputs

This framework does not authorize inventing the following:

- portrait photography;
- ByteDance cover, PDF, or native case evidence;
- Meeting screenshots, video, captions, or outcome data;
- Visual Archive project names, images, roles, years, or URLs;
- resume files;
- email, LinkedIn, WeChat ID, or WeChat QR.

These inputs replace Draft states later without changing the approved homepage
information architecture.
