# ConvoAI 中文案例叙事重构规格

日期：2026-07-22

## 目标

把当前以素材分组和概念口号为主的中文页面，重构为一篇由真实产品场景、实时交互矛盾和关键设计决策逐步推进的案例。页面需要让招聘经理在快速浏览时理解产品、职责和五个核心决策，在深入阅读时看到跨端差异、系统约束与设计取舍。

## 证据边界

- **事实：** 已提供 9 段 App 与 7 段 Web 产品录屏，以及 Figma 参考画面。
- **事实：** 录屏可支持登录、启动、连续对话、字幕与摄像头、声纹、硬件、数字人、语音打断和实时数据等界面与交互描述。
- **事实：** `Designer-reported: 独立负责产品设计；产品已正式上线。`
- **假设：** “让实时状态可理解、可介入、可恢复”是作品集的叙事解释，不写成用户研究结论。
- **缺口：** 没有可验证的业务指标、用户研究结论、转化或满意度结果。
- **边界：** 不把 RTC/RTM 基础设施、模型、数字人驱动或生产代码写成个人设计产出。

## 方案比较

### 方案 A：仅润色现有中文

保留七章和当前媒体分组，只修改病句。成本最低，但无法解决“建立就绪”承载七段视频、字幕视频被归入打断、结尾被证据说明占据等结构问题。

### 方案 B：按 App / Web 分成两条故事线

先讲完整 App，再讲完整 Web。素材归属最清楚，但同一设计问题会被重复解释，跨端设计判断会退化为功能清单。

### 方案 C：按设计决策组织，跨端素材作为证据

先建立背景、挑战与状态模型，再以建立会话信心、保留用户控制、确认参与者、增强临场感和定位实时问题五个决策展开。App 与 Web 只在能够解释同一个问题时同章出现。

**采用方案 C。** 用户已于 2026-07-22 确认该方向。

## 新章节结构

### 00 项目背景

回答：ConvoAI 是什么、服务什么场景、设计师负责什么？

- 说明它是面向客户体验演示的 1 对 1 实时 AI 对话产品，覆盖 App 与 Web。
- 职责与上线状态保留 `Designer-reported`。
- 登录和页面结构只承担产品定位，不作为关键决策。

### 01 设计挑战与状态模型

回答：实时 AI 通话为什么比文字聊天更难理解？

- 用连接、聆听、生成、播放解释不可见系统如何连续变化。
- 字幕、摄像头、球体和数字人是状态反馈渠道，不与底层阶段并列。
- 三条策略“可感知、可介入、可恢复”在问题之后出现，表现为设计回应而非先验口号。
- 状态链作为本章系统图：Connecting → Ready → Listening → Thinking → Speaking → Interrupted → Recovery。

### 02 建立会话信心

回答：用户如何判断何时可以开口，以及会话何时真正开始或结束？

- 关键 App 证据：`app-conversation-start`。
- 关键 Web 证据：`web-preflight-layout`、`web-join-exit`。
- `app-login`、`app-structure`、`web-login`、`web-preflight` 降为背景素材，可保留在同章的辅助列表中，但不与关键决策获得同等文字权重。
- 跨端结论：App 压缩准备步骤，Web 展开配置；两端共享状态语义，而非共享同一布局。

### 03 保留用户控制

回答：用户打断 AI 后，系统如何停止旧回答并接住新意图？

- 先用 `web-conversation` 建立正常连续对话，再用 `web-interrupt` 展示打断与恢复。
- `app-caption-camera` 用于证明声音、字幕、球体、视频和控件需要同步更新，不写成 App 打断证据。
- 重点解释旧回答停止、被打断内容保留、Thinking 转换与新回答恢复。

### 04 确认谁在参与

回答：系统如何明确正在听谁，以及哪些设备参与当前对话？

- `app-voiceprint-lock` 是本章主证据，单独解释 Off、Seamless、Personalized 的便利、控制和录入成本取舍。
- `app-hardware-device` 是连接准备的辅助证据。
- `app-profile-settings` 作为个人身份上下文，不包装成高风险设计决策。
- 不再用“可信边界”统一抬高三个功能。

### 05 增强临场感

回答：数字人增强临场感后，哪些实时问题被放大？

- `app-avatar-select` 建立互动预期。
- `app-avatar-interaction` 展示连接、话轮、摄像头和画中画。
- 重点解释音画同步、延迟可见性和视觉焦点，不把数字人写成视觉换肤功能。

### 06 定位实时问题

回答：当用户觉得响应慢时，产品和工程如何定位问题？

- `web-realtime-data` 是唯一主证据。
- 解释 E2E、RTC、ASR、LLM、TTS 与 Voiceprint 各自对应的可观察问题。
- “可定位”是设计意图，不声称已提升调试效率。

### 07 交付与反思

回答：最终交付了什么，这个项目沉淀了什么判断？

- 先陈述跨端产品设计范围与正式上线状态，保留归因。
- 单独说明没有可验证的业务指标，不把证据免责声明写成主结论。
- 反思落在一个具体判断：实时 AI 的控制感来自状态、可用行动和恢复路径持续一致，而不是增加控件。
- “9 段 App 与 7 段 Web 视频完整保留”放在末尾证据注释，不承担情绪收尾。

## 中文写作规则

- 每章按照“场景 → 矛盾 → 设计动作 → 预期作用”推进。
- 优先使用具体主语：用户、AI、界面、App、Web、系统；减少“核心设计对象”“状态真相”“可信边界”等抽象名词。
- 只在真正存在对比时使用“不是……而是……”，避免连续使用口号式反转。
- 避免英文句法直译，例如“推动声音与文本变化”“从选择预期过渡”“保留状态理解”。
- `Agent`、`RTC`、`ASR`、`LLM`、`TTS` 等产品或技术术语可以保留；章节标题和 CPDI 标签使用自然中文。
- 无结果证据时统一使用“设计意图是”或“预期帮助”，不使用“提升”“改善”“确保”等已验证结果语气。
- `Designer-reported` 只在职责、上线与影响边界出现，不在每章重复。

## 媒体卡片规则

- 每段视频继续完整播放；9 段 App 不裁切、不加速、不跳帧、不节选、不局部循环。
- 播放列表保留 Context / Problem / Decision / Impact 数据结构，中文界面显示为“场景 / 问题 / 设计 / 作用”。
- 低信息量背景视频保留简短 CPDI，高风险决策视频提供完整取舍。
- `app-caption-camera` 的说明改为多通道同步，不再借用“完整因果链”等翻译式表达。
- 视觉叠加舞台继续保持 Web 为主、App 手机壳位于右前景、最大 4° 透视。

- 共享顶部导航与左侧导航的色彩系统。
- 视频源文件、时长与播放速度约束。
- 既有产品色：案例内容使用 cyan / blue / coral，导航保留全局 iris。

## 验收标准

- 中文页面按背景、挑战、五个决策、交付反思自然推进。
- 五个核心决策全部保留：建立会话信心、打断与控制、声纹与参与者、数字人、实时可观测性。
- App 与 Web 的组合由共同问题解释，不再机械并列。
- 页面不出现 Gap / 缺口章节。
- 职责和上线状态仍标记 `Designer-reported`。
- 所有中文章节标题与正文通过逐句口语化检查，无明显英文直译语序。
- 9 段 App 和 7 段 Web 视频仍可访问，App 文件与时长不变。
- 单元测试、组件测试和 ConvoAI 端到端关键路径通过。
- 桌面与移动端截图确认导航、标题、视频和 CPDI 不重叠、不溢出。
