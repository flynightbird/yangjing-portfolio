# ConvoAI 中文案例叙事重构规格

日期：2026-07-22

## 目标

把当前以素材分组和概念口号为主的中文页面，重构为一篇由真实产品场景、App 设计完整度和实时交互问题逐步推进的案例。页面需要让招聘经理在快速浏览时理解产品、职责、App 结构与四个实时体验主题，在深入阅读时看到跨端差异、系统约束与设计取舍。

## 证据边界

- **事实：** 已提供 9 段 App 与 7 段 Web 产品录屏，以及 Figma 参考画面。
- **事实：** 录屏可支持登录、启动、连续对话、字幕与摄像头、声纹、硬件、数字人、语音打断和实时数据等界面与交互描述。
- **事实：** `Designer-reported: 独立负责产品设计；产品已正式上线。`
- **假设：** “让实时状态可理解、可介入、可恢复”是作品集的叙事解释，不写成用户研究结论。
- **缺口：** 没有可验证的业务指标、用户研究结论、转化或满意度结果。
- **边界：** 不把 RTC/RTM 基础设施、模型、数字人驱动或生产代码写成个人设计产出。

## 方案比较

### 方案 A：仅润色现有中文

保留七章和当前媒体分组，只修改病句。成本最低，但无法解决“建立就绪”承载七段视频、字幕视频被归入打断、结尾被证据说明占据等结构问题。

### 方案 B：按 App / Web 分成两条故事线

先讲完整 App，再讲完整 Web。素材归属最清楚，但同一设计问题会被重复解释，跨端设计判断会退化为功能清单。

### 方案 C：按设计决策组织，跨端素材作为证据

先建立背景、挑战与状态模型，再单独展示 App 产品结构，随后以开始实时对话、对话控制、数字人互动和实时链路四个主题展开。App 与 Web 只在能够解释同一个问题时同章出现。

**采用方案 C。** 用户已于 2026-07-22 确认该方向。

## 新章节结构

### 00 项目背景与设计挑战

回答：ConvoAI 是什么、服务什么场景、设计师负责什么，以及实时 AI 通话为什么比文字聊天更难理解？

- 说明它是面向客户体验演示的 1 对 1 实时 AI 对话产品，覆盖 App 与 Web。
- 职责与上线状态保留 `Designer-reported`。
- 用连接、聆听、生成、播放解释不可见系统如何连续变化。
- 字幕、摄像头、球体和数字人是状态反馈渠道，不与底层阶段并列。
- 三条策略“可感知、可介入、可恢复”在问题之后出现，表现为设计回应而非先验口号。
- 状态链作为本章系统图：Connecting → Ready → Listening → Thinking → Speaking → Interrupted → Recovery。

### 01 App 产品结构与 UI 交互

回答：除了核心通话，App 如何组织入口、首页、个人设置与设备连接？

- 这是产品设计与交互完整度展示，不包装成高风险战略决策。
- 使用参考 Sequence 的双栏切换结构：左侧为内容导航与说明，右侧为固定手机媒体舞台。
- 这组双栏位于详情模板的正文栏内。页面外层仍使用统一的顶部导航、左侧章节导航、内容栅格与响应式折叠规则，不为 ConvoAI 新建另一套详情框架。
- 外层左侧章节导航负责跨章节定位；App 章节内的左列只负责四个 App 场景，不复用章节导航的视觉重量，避免同一页面出现两组互相竞争的主导航。
- 四个切换项依次为 `app-login`、`app-structure`、`app-profile-settings`、`app-hardware-device`。
- 桌面端以纵向滚动作为主要切换方式：左侧四个条目按页面阅读顺序展开，右侧手机在本章范围内保持固定；条目经过统一激活线时，右侧固定位置替换为对应完整录屏。
- 当前项呈现明确激活态；点击或键盘激活其他条目时，页面滚动到对应内容，右侧媒体随滚动位置同步更新。点击是快速跳转方式，不建立一套独立于滚动位置的选中状态。
- 右侧只允许一段视频播放。激活视频从头开始，以 1 倍速、静音、完整时长循环；离开激活态后立即暂停并复位。
- 用户主动开启声音后，不因自动切换播放其他带声视频。
- 桌面端右侧媒体只在本章滚动范围内保持 sticky，进入下一章后正常离场；移动端取消 sticky，改为单列手风琴，当前视频放在对应说明下方。
- `prefers-reduced-motion` 下不自动播放，显示 poster 与标准播放控件。

#### App 媒体卡片

- 右侧媒体舞台必须置于独立媒体卡片中，卡片圆角固定为 `20px`；圆角属于本章媒体展示，不改变详情模板中的通用卡片或导航圆角。
- 媒体卡片保持稳定宽高和固定手机锚点。场景切换时只替换卡片底色与手机内视频，手机外壳不得发生尺寸、位置或透视跳动。
- 四个场景使用不同的低饱和莫兰迪底色：登录为雾蓝 `#B8C7C9`，产品结构为灰绿 `#BEC4B3`，个人设置为灰粉 `#CFB9B6`，硬件设备为灰褐 `#C6BDAD`。
- 底色只承担区分场景和衬托深色手机的作用，不编码产品状态，不延伸到顶部导航、左侧章节导航、正文标题或 CPDI 标签。
- 卡片不使用渐变、发光或额外装饰图形。手机外壳是媒体载体，不再嵌套第二层装饰卡片。
- 普通动效设置下，底色与媒体可使用克制的淡入切换；`prefers-reduced-motion` 下即时替换，不做颜色过渡或位移动画。

#### App 产品结构滚动交互

- 桌面端为每个左侧条目保留至少 `60vh` 的滚动区间，正文较长时允许自然撑高，确保四个状态都有稳定、可阅读的停留范围。
- 视口高度约 `42%` 处作为唯一激活线。当前项定义为最后一个顶部已经越过激活线的条目；在第一个条目之前固定为第一项，在最后一个条目之后固定为最后一项。
- 使用 `IntersectionObserver` 观察四个条目的激活标记，并通过窄 `rootMargin` 区间模拟激活线。实现必须保持单一 active id；不得因两个条目同时出现在视口中而来回切换。
- 向下滚动依次激活后续条目，向上滚动按相反顺序恢复之前条目。激活逻辑由条目位置决定，不依赖滚动方向猜测。
- 左侧条目标题使用可聚焦按钮。点击、Enter 或 Space 后，将对应条目顶部滚动到激活线位置；普通动效设置下使用平滑滚动，`prefers-reduced-motion` 下立即跳转。
- 激活项变化时，先暂停并将上一段视频复位到 `currentTime = 0`，再加载或播放新项。一次只允许当前视频自动播放，循环单位始终是完整源视频。
- sticky 容器由 App 产品结构章节自身约束，不能遮挡顶部导航，也不能延伸进入“开始一段实时对话”章节。
- 移动端不使用基于视口的滚动激活线。四项按单列手风琴呈现，点击标题展开对应说明和完整视频；切换时仍执行单视频播放与上一段复位规则。
- `prefers-reduced-motion` 不取消内容切换，但关闭自动播放、平滑滚动和媒体过渡；当前项显示 poster 与标准播放控件。

### 02 从进入产品，到开始一段实时对话

回答：App 与 Web 如何完成进入、准备、连接、加入与退出？

- 关键 App 证据：`app-conversation-start`。
- 关键 Web 证据：`web-preflight-layout`、`web-join-exit`。
- `web-login` 与 `web-preflight` 作为 Web 流程的辅助证据。
- 跨端结论：App 压缩准备步骤，Web 展开配置；两端共享状态语义，而非共享同一布局。
- 不使用“建立信心”等无法被录屏直接证明的心理结果表述。

### 03 对话中的控制权

回答：用户如何改变 AI 的话轮，以及哪些声音能够触发 Agent？

- 先用 `web-conversation` 建立正常连续对话，再用 `web-interrupt` 展示打断与恢复。
- `app-caption-camera` 用于证明声音、字幕、球体、视频和控件需要同步更新，不写成 App 打断证据。
- 重点解释旧回答停止、被打断内容保留、Thinking 转换与新回答恢复。
- `app-voiceprint-lock` 作为控制权的第二层证据：Off、Seamless、Personalized 体现便利性、身份控制与录入成本的取舍。
- 本章只讨论用户层行动与反馈，不出现 E2E、RTC、ASR、LLM、TTS 等系统指标。

### 04 数字人互动

回答：用户如何从选择数字人进入实时互动？

- `app-avatar-select` 建立互动预期。
- `app-avatar-interaction` 展示连接、话轮、摄像头和画中画。
- 重点解释音画同步、延迟可见性和视觉焦点，不把数字人写成视觉换肤功能。
- 两段 App 完整录屏在同一媒体区域平铺，不使用 Web/App 透视叠加。
- 桌面端双列并排、静音、1 倍速、完整时长循环；移动端纵向排列。
- 每段视频保留独立标题与简短说明，说明选择与互动的前后关系。

### 05 一次回答背后的实时链路

回答：用户听到一次回答时，媒体、识别、生成与合成过程如何同时发生？

- `web-realtime-data` 是唯一主证据。
- 将实时数据放在正在发生的对话旁边，而不是呈现为脱离体验的独立监控面板。
- 解释 E2E、RTC、ASR、LLM、TTS 与 Voiceprint 分别代表的处理阶段或状态。
- 本章面向演示与技术视角，连接体验现场与系统过程；不再重复“用户能否打断、能否控制”的叙述。
- 不使用“可定位”“提升调试效率”等结果表述，也不声称该设计改善了系统性能。

### 06 交付范围

回答：最终交付了什么，现有证据能够支持哪些结论？

- 先陈述跨端产品设计范围与正式上线状态，保留归因。
- 单独说明没有可验证的业务指标，不把证据免责声明写成主结论。
- “9 段 App 与 7 段 Web 视频完整保留”放在末尾证据注释，不承担情绪收尾。
- 公开案例删除反思、价值升华和“这个项目让我更确定”等主观结尾；内部蓝图与面试准备仍保留复盘问题，不显示在页面中。

## 中文写作规则

- 每章按照“场景 → 矛盾 → 设计动作 → 预期作用”推进。
- 优先使用具体主语：用户、AI、界面、App、Web、系统；减少“核心设计对象”“状态真相”“可信边界”等抽象名词。
- 只在真正存在对比时使用“不是……而是……”，避免连续使用口号式反转。
- 避免英文句法直译，例如“推动声音与文本变化”“从选择预期过渡”“保留状态理解”。
- `Agent`、`RTC`、`ASR`、`LLM`、`TTS` 等产品或技术术语可以保留；章节标题和 CPDI 标签使用自然中文。
- 无结果证据时统一使用“设计意图是”或“预期帮助”，不使用“提升”“改善”“确保”等已验证结果语气。
- `Designer-reported` 只在职责、上线与影响边界出现，不在每章重复。

## 媒体卡片规则

- 每段视频继续完整播放；9 段 App 不裁切、不加速、不跳帧、不节选、不局部循环。
- App 产品结构与数字人展示可以循环播放，但循环单位必须是完整源视频，不得只循环局部片段。
- 播放列表保留 Context / Problem / Decision / Impact 数据结构，中文界面显示为“场景 / 问题 / 设计 / 作用”。
- 低信息量背景视频保留简短 CPDI，高风险决策视频提供完整取舍。
- `app-caption-camera` 的说明改为多通道同步，不再借用“完整因果链”等翻译式表达。
- 视觉叠加舞台继续保持 Web 为主、App 手机壳位于右前景、最大 4° 透视。
- Sequence 式切换只用于 App 产品结构章节；其他章节不重复使用，以免页面交互单一。
- Sequence 式展示的“切换”仅改变当前激活的完整源视频，不裁切、跳过或改写任何视频内容。
- App 产品结构的四张媒体卡片统一使用 `20px` 圆角和稳定尺寸，以四种指定莫兰迪底色区分；这些颜色不改变全局导航主题。
- 数字人章节使用无透视的双 App 平铺，与跨端叠加舞台形成明确差异。

## 详情模板约束

- 顶部导航继续使用站点共享 `SiteHeader`，高度、层级、交互与全站详情页一致。
- 左侧章节导航继续使用共享 `ChapterNav`，沿用模板的 sticky 位置、当前章节状态、键盘焦点样式和窄屏折叠方式。
- 详情页外层继续使用既有 `rail + case` 栅格、内容 gutter、章节间距与断点；App 章节的 Sequence 双栏只能存在于 `case` 正文栏内部。
- 全局导航在深浅章节间的色值切换继续由详情模板控制，保留 iris 强调色。莫兰迪媒体卡片不得驱动或覆盖导航色值。
- App 章节内的场景列表使用局部样式和局部 active id，不注册为全局章节，不出现在详情页章节导航中。
- 桌面端同时出现外层章节导航和内层场景列表时，必须通过字号、缩进、编号与激活样式建立清晰层级；移动端外层章节导航先按模板折叠，内层场景列表再转为正文手风琴。

## 验收标准

- 中文页面按背景与挑战、App 产品结构、开始实时对话、对话控制、数字人互动、实时链路、交付范围自然推进。
- 页面不再出现“建立会话信心”“确认谁在参与”“定位实时问题”或“交付与反思”等旧章节。
- App 与 Web 的组合由共同问题解释，不再机械并列。
- App 产品结构在桌面端由纵向滚动驱动左侧激活项与右侧固定手机同步切换；向上滚动能够按相反顺序恢复，且 sticky 不越过本章边界。
- 点击或键盘激活左侧条目会滚动到对应内容，不产生与滚动位置冲突的第二套选中状态；移动端退化为可访问的单列手风琴。
- App 产品结构右侧使用稳定尺寸的 `20px` 圆角媒体卡片；四个场景按规格切换不同莫兰迪底色，手机位置与尺寸不跳动。
- 顶部导航、左侧章节导航、页面栅格和断点行为沿用详情模板；App 内部场景列表不进入全局章节导航，莫兰迪色不影响导航色值。
- App 产品结构切换时只有当前视频播放，且完整视频以 1 倍速循环；reduced-motion 下不自动播放。
- 数字人两个 App 场景桌面并排、移动端纵向排列，不使用透视叠加。
- “对话中的控制权”只讲用户层；“一次回答背后的实时链路”只讲演示与技术视角，媒体和文案不重复。
- 公开页面结尾只呈现交付范围与证据注释，不包含反思或价值升华；内部面试材料不受此限制。
- 页面不出现 Gap / 缺口章节。
- 职责和上线状态仍标记 `Designer-reported`。
- 所有中文章节标题与正文通过逐句口语化检查，无明显英文直译语序。
- 9 段 App 和 7 段 Web 视频仍可访问，App 文件与时长不变。
- 单元测试、组件测试和 ConvoAI 端到端关键路径通过。
- 桌面与移动端截图确认导航、标题、视频和 CPDI 不重叠、不溢出。
