02 / CONTEXT & ROLE

从技术能力集合，
到呼叫场景产品

企业要完成一次 AI 呼叫，必须跨越文档、API、模型与声音供应商、号码配置和任务系统。能力存在，但客户难以独立完成接入，更难在发布前判断 AI 是否真的可用。

呼叫场景成为团队核心主推方向后，原有对话式 AI 引擎演进为 AI Studio；与呼叫相关的能力被抽离为独立的智能呼叫中心，通过左侧产品切换器进入。这不是功能合并，而是依据任务闭环重新划分产品边界。

04 / DESIGN DECISION 01

重构产品边界与任务路径

从“先理解资源结构”转向“先完成一个呼叫任务”

早期版本把模型、ASR、TTS、MCP 等资源直接暴露为主要入口。对熟悉底层能力的内部团队有效，却要求企业客户先理解系统结构，才能开始创建智能体。

我将呼叫能力单独成产品，以智能体、号码、外呼任务、历史记录为主导航；公共资源下沉为支撑能力，并通过产品切换器保留与 AI Studio 的关系。

DECISION / 以任务组织能力

  1. 先选产品通用智能体与呼叫运营分开
  2. 再建智能体模板、基础配置与高级配置逐层展开
  3. 资源按需出现在任务中引用，而不是要求用户先配置全部资源

05 / DESIGN DECISION 02 · CORE

让 AI 在发布前
可见、可测、可控

AI 配置的难点不是字段多，而是输入与行为之间缺少确定反馈。用户不知道 Prompt、声音或模型调整后会发生什么，也无法区分配置错误、连接问题与模型表现。

已连接 实时转写 待调整
INPUT

调整配置

Prompt、开场白、模型、语音与通话参数。

RUN

发起 Preview

在真实通话形态下验证，而不是只看静态表单。

OBSERVE

观察运行状态

连接、时长、转写、AI 回复与异常信息同步出现。

ITERATE

定位并调整

把模糊的“效果不好”转成可判断的修改依据。

06 / DESIGN DECISION 03

从 AI Demo 到呼叫运营闭环

发布不是终点，结果可诊断才是产品化

智能体可以说话，并不等于企业可以运营它。真实呼叫还需要号码、批量任务、发布检查、失败原因、历史筛选、信息提取与数据分析。

我把发布后的工作纳入同一任务链，让运营人员能够从任务创建进入执行，再从结果回到诊断，而不是在多个系统之间寻找答案。

DECISION / 按诊断频率分层

保留时间、呼叫类型和搜索作为高频入口；把长尾条件收进“更多设置”，并让已选条件以可撤销标签持续可见。

发布检查号码管理外呼任务失败原因信息提取数据分析

07 / SYSTEM & DELIVERY

从 Design System
到可执行的工程交付

FOUNDATIONColor · Type · Spacing · Radius
SEMANTICSurface · Content · Border · Status
COMPONENTInput · Select · Table · Navigation
ConnectedFailed

我从 0 到 1 搭建设计系统，将组件、状态与各类 Token 沉淀到产品仓库。随后拉取真实前端代码，复用仓库内的 Design System，通过 AI 辅助编码完成独立高保真前端原型。

它不是生产代码，而是一份可运行的交付资料：前端工程师可以直接核对组件、状态和响应式规则，减少从设计稿反推实现细节的成本。

01设计规格布局、状态与响应式规则
02AI 辅助原型基于真实仓库验证实现
03工程生产页面由前端工程师完成正式代码

责任边界 该高保真前端原型用于验证设计方案并辅助工程交付，未合入生产环境；正式生产代码由前端工程师完成。

公开证据边界 本案例仅使用已脱敏的产品界面与可确认的过程事实；AI 辅助原型未合入生产，因此不以工程效率或还原度数据作为结果结论。

08 / OUTCOME & LEARNINGS

阶段结果、证据与反思

V1.0 已完成交付并进入有限客户灰度阶段，已向少量企业客户开放，但尚未规模化推广。因此本案例不将灰度状态解释为业务增长结果，而聚焦已建立的产品闭环、设计决策与下一阶段验证基础。

DELIVERED / 已交付

完整产品闭环

独立产品边界、配置与 Preview、发布与呼叫运营、Design System 和工程交付资料。

OBSERVED / 已观察

有限范围验证

产品、研发、算法、音视频与 QA 内部评审，以及少量客户灰度开放。

NOT YET VALIDATED

尚未规模验证

客户任务完成率、Preview 到发布转化、留存与业务增长均没有足够样本。

WHAT I LEARNED

AI 产品的确定性，来自可观察的反馈

配置项越专业，越需要把系统状态、运行结果与错误原因前置。Preview 不是附加功能，而是用户建立控制感的核心基础设施。
