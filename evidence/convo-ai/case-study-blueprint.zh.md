## 2. 项目定位

**建议（Recommendation）：**

> `Designer-reported: ConvoAI 面向客户体验演示。` ConvoAI 是一个跨 App 与 Web 的 1 对 1 实时 AI 对话产品。案例围绕用户如何进入会话、理解 AI 的当前状态、改变话轮，并看到一次回答经过的处理环节来组织可见证据。

**状态模型（案例解释，非已验证规范）：**Connecting → Ready → Listening → Thinking → Speaking → Interrupted → Recovery。进入、准备、加入和退出属于用户流程，不替代这条状态模型。

**交付范围：**`Designer-reported: 独立负责 ConvoAI 产品设计；产品已正式上线。` 现有材料没有经佐证的研究或业务结果。16 段录屏只是在结尾用于核对可见界面范围，不是设计交付本身。

## 4. 视觉方向

**建议（Recommendation）：**以真实产品界面、连续状态和简洁注释说明实时对话；不使用泛 AI 科幻图形替代产品证据。

- 外层的 `SiteHeader`、`ChapterNav`、rail + case grid、断点和 iris 导航主题遵循共享详情页模板。章节内的局部视觉不改变全站导航语义。
- Hero 以 Web 界面为主，使用提供的 Figma 界面 `/images/convo-ai/figma/web-ready.png`，App 手机置于右侧前景，透视最多 4 度。图注和交付范围面板必须说明“Figma 界面证据”。
- 字体继承共享详情页模板的标题与正文字体层级。等宽字只用于状态、媒体索引、技术缩写和数值，不能覆盖正文。
- App 产品结构是正文内的局部 Sequence 式版块：左侧滚动叙述，右侧固定、尺寸稳定的手机。右侧媒体卡片圆角必须为 20px；四张卡片依次使用登录雾蓝 `#B8C7C9`、结构灰绿 `#BEC4B3`、个人灰粉 `#CFB9B6`、硬件灰褐 `#C6BDAD`。这些颜色不驱动导航。
- 数字人采用两段 App 录屏并列的平面双屏，移动端上下堆叠；不做透视叠加，也不以装饰关系替代“选择 -> 互动”的前后关系。
- 实时链路图将 E2E、RTC、ASR、LLM、TTS、Voiceprint 与正在进行的对话并置，说明媒体、识别、生成和合成的先后关系；它不声称改善调试或系统性能。
- 叙事中的状态只使用 Connecting → Ready → Listening → Thinking → Speaking → Interrupted → Recovery。说明文字只解释当前界面的 Context、Problem、Decision、Impact，避免随机截图墙。

### 媒体约束

1. 9 段 App 视频全部保持完整：不得裁切（no trim）、加速（no speedup）、跳帧（no frame skip）、制作节选（no excerpt）或局部循环（no partial loop）。
2. App 产品结构与数字人可用完整源片静音、1x 循环；不改变时长、帧序或内容。数字人双屏可以同时完整循环。
3. 减弱动态时显示海报帧并提供手动播放；在正常动态模式下仅播放当前激活的展示视频（only active showcase video），数字人双屏是唯一可同时完整循环的例外。
4. 7 段 Web 视频都必须保持可到达；其中实时数据录屏完整呈现，不裁成技术片段。
5. 这是发布要求。Task7 尚未完成浏览器运行时回放核验，不能把完整播放、减弱动态和可到达性写成已验证事实。

## 7. 3 分钟叙事

> ConvoAI 是一个跨 App 与 Web 的 1 对 1 实时 AI 对话产品。`Designer-reported: 产品用于面向客户的体验演示；我独立负责产品设计；产品已正式上线。` 这些是设计师的归属性陈述，现有材料没有独立的发布、客户或结果记录。
>
> 这个案例的难点不在于做一个有输入框的聊天页，而在于让一段实时通话保持可理解。用户会经历 Connecting、Ready、Listening、Thinking、Speaking、Interrupted 和 Recovery。这里有一个工作解读，不是用户研究结论：录屏显示，当连接、语音、字幕、摄像头和 AI 回答同时改变时，界面需要持续告诉人“现在发生了什么”和“我还能做什么”。
>
> 因此我会把设计策略讲成三件事：让状态可感知、让用户可介入、让过程可恢复。第一，App 在启动时依次给出权限、连接和 Ready 反馈；这是对“何时可以开口”的工作解读，预期作用是为等待过程提供可见线索。第二，Web 对话把转录、动感球、稳定控制区和可折叠 Agent rail 放在一起；信息密度更高，但用户不必为了看状态或切换 Agent 离开对话。第三，语音打断保留 Interrupted 内容，转入 Thinking 后再给出新回答；它避免把旧内容直接抹掉，也让用户看见 AI 正在重新处理。声纹锁定则把“哪些声音可以触发 Agent”变成 Off、Seamless 和 Personalized 三种明确选择。
>
> 数字人和实时数据不是附加展示：前者把角色选择连接到 Agent 配置和实际互动，后者把一次回答旁的媒体、识别、生成和合成阶段展示出来。交付范围先按归属说明产品结构、交互和跨端体验设计；其次明确没有研究、指标或结果可证明用户或业务成效；最后才以提供的 Figma 图和 16 段录屏作为可见界面证据。这个案例能展示对实时交互的产品判断，但不把它包装成未经证实的影响故事。
>
> 仅供内部面试反思，不进入公开页面：下一步会先测试用户能否理解状态、成功打断、设置声纹和完成设备连接，并补齐设计决策和协作记录；在此之前，这些都只能是待验证的计划。

### 内部优先级与取舍

1. **决策理由与取舍：**先补齐关键界面的产品判断、替代方案和预期验证。没有这些材料，精致的媒体只能说明“做了什么”。
2. **协作与职责证据：**再确认协作者、技术约束、冲突和发布状态，避免把 `Designer-reported` 误写成独立成果。
3. **逐屏产品判断：**用 CPDI 复核每一屏的 Context、Problem、Decision、Impact，删除无法说明产品选择的画面。
4. **研究与结果：**补充用户、客户或业务证据，或持续明确其缺失。
5. **媒体与模板：**最后检查 Figma 披露、完整回放、可达性、Sequence 稳定性和数字人移动端堆叠；这些优化不应替代前四项。
