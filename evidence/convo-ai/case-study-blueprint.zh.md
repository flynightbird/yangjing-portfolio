## 2. 项目定位

**建议（Recommendation）：**将案例定位为：

> ConvoAI 是面向客户体验演示的跨端 1 对 1 AI 对话产品；设计的核心不是展示更多 AI 能力，而是把不可见、随时变化的实时通话状态，变成人能够理解、打断并继续的自然互动。

**建议（Recommendation）：**不要使用“视频对话 + 字幕 + 动感球 + 数字人”作为主叙事；这是功能清单，不是设计判断。

**建议（Recommendation）：**暂用作用域行：“独立负责产品设计（Designer-reported）；产品已正式上线（Designer-reported）。待项目记录、生产录屏或公开发布信息完成佐证。”

## 4. 视觉方向

**建议（Recommendation）：**案例视觉语言应表现“实时互动的节奏与状态”，而不是泛 AI 科幻感。

- 层级：首屏用一个真实通话瞬间 + 一句核心转变；章节在“系统概览”与“高信息局部”间切换。
- 字体：延续作品集的干净无衬线体；标题用中等尺度，状态标注使用等宽数字或紧凑标签。
- 颜色：中性背景 + 一个产品主色；颜色只用于区分 Listening / Thinking / Speaking / Interrupted / Recovery，不做装饰渐变。
- 网格：桌面端使用 12 列，文字阅读区控制在 5–6 列；手机录屏以稳定高宽比居中，不和 Web 画面挑战视觉重心。
- 密度：录屏周围只保留一层 CPDI 标注；技术状态放入独立系统图，不堆在界面上。
- 图示语言：实线代表录屏可验证的触发与反馈，虚线代表尚未佐证的 RTC / RTM / AI 依赖。
- 界面处理：视频优先保留连续状态；静帧只用于放大关键反馈、比较跨端或标注取舍。

## 6. 屏幕级 CPDI

> 当前尚未观看原始录屏，以下是用于筛选素材的临时 CPDI，不是已证实的产品理由。

### App 01：登录页

- Context — 事实（Fact）：`Designer-reported` App 录屏包含登录页。
- Problem — 假设（Assumption）：登录是必要入口，但不是案例最有区分度的实时对话问题。
- Decision — 建议（Recommendation）：只用一帧或 2–3 秒进入动作建立产品起点，并与 Web 登录并列；除非存在特殊身份或权限判断，不单独成章。
- Impact — 假设（Assumption）：预期让评审快速理解产品进入路径。缺口（Gap）：需原片确认信息与设计价值。

### App 02：对话互动

- Context — 事实（Fact）：`Designer-reported` 用户在 App 中与 AI 进行 1 对 1 互动。
- Problem — 假设（Assumption）：AI 的聆听、思考、回答和被打断都缺少真人对话中的自然线索。
- Decision — 建议（Recommendation）：将这条录屏作为核心证据，用时间轴标注话轮变化，并检查球体、字幕、声音与控件是否表达同一状态。
- Impact — 假设（Assumption）：意图是让用户理解话轮并保持介入权。缺口（Gap）：需原片与使用反馈佐证。

### App 03：字幕 / 摄像头开启后的互动

- Context — 事实（Fact）：`Designer-reported` 录屏展示字幕与摄像头开启后的完整互动，内容较多且不适合删减。
- Problem — 假设（Assumption）：文字、视频、球体、AI 输出与通话控件会在有限屏幕上竞争注意力，剪短录屏又可能破坏实时因果链。
- Decision — 建议（Recommendation）：保留完整录屏，增加可跳转时间点（开启摄像头 / 开启字幕 / 互动状态变化 / 结束），再用 3–4 帧静态摘要解释信息优先级。
- Impact — 假设（Assumption）：预期同时保留真实性与可扫读性。缺口（Gap）：需原片时长、关键时间点和字幕可读性检查。

### App 04：个人设置页

- Context — 事实（Fact）：`Designer-reported` App 录屏包含个人设置页。
- Problem — 假设（Assumption）：对话偏好、设备或账户信息若只按技术资源分类，用户难以预判它们对实际通话的影响。
- Decision — 建议（Recommendation）：用一幅可读的全页图解释信息架构，只标出与后续对话或硬件绑定直接相关的设置。
- Impact — 假设（Assumption）：预期让偏好与对话结果的关系更清晰。缺口（Gap）：需视频内容和设置分类佐证。

### App 05：个人设置操作

- Context — 事实（Fact）：`Designer-reported` 录屏展示用户实际修改个人设置。
- Problem — 假设（Assumption）：实时产品中的某些偏好可能立即影响通话，某些只在下次会话生效，如果缺少反馈就会产生状态不确定。
- Decision — 建议（Recommendation）：保留一个完整的“修改 → 确认 → 生效”任务，解释即时生效、保存和失败反馈的取舍。
- Impact — 假设（Assumption）：预期降低对设置是否成功的不确定。缺口（Gap）：需原片确认生效方式。

### App 06：绑定硬件设备

- Context — 事实（Fact）：`Designer-reported` App 录屏展示绑定外部硬件设备。
- Problem — 假设（Assumption）：发现、连接、授权和可用状态分属不同系统阶段，但用户只关心设备是否能用于下一次对话。
- Decision — 建议（Recommendation）：用连续序列展示发现、选择、绑定和就绪，并将它与个人设置合并为“对话准备”章节。
- Impact — 假设（Assumption）：预期让设备状态与下一步行动更可判断。缺口（Gap）：需原片与异常状态佐证。

### App 07：数字人选择

- Context — 事实（Fact）：`Designer-reported` 用户可以在进入数字人对话前进行选择。
- Problem — 假设（Assumption）：如果选择只呈现视觉差异，用户无法理解它是否会影响声音、性格、能力或互动方式。
- Decision — 建议（Recommendation）：将选择页与后续互动放在同一组 CPDI 中，解释用户在选择时获得的预期与真实互动的对应关系。
- Impact — 假设（Assumption）：预期使数字人选择成为有意义的对话决策，而不是换肤。缺口（Gap）：需原片与角色属性佐证。

### App 08：数字人互动

- Context — 事实（Fact）：`Designer-reported` 录屏展示与数字人的实时互动。
- Problem — 假设（Assumption）：数字人提升临场感，也放大音画同步、凝视、字幕竞争、延迟与打断反馈的问题。
- Decision — 建议（Recommendation）：保留一段连续话轮，标注嘴型 / 声音、字幕、控制和角色反馈之间的优先级；与数字人选择形成“预期 → 体验”序列。
- Impact — 假设（Assumption）：意图是在更强的社会临场感中保留理解与控制。缺口（Gap）：需用户反馈或任务证据。

### Web 01：登录页

- Context — 事实（Fact）：`Designer-reported` Web 录屏包含登录页。
- Problem — 假设（Assumption）：Web 入口需要保留和 App 一致的产品认知，又需要适应浏览器的身份、权限与窗口环境。
- Decision — 建议（Recommendation）：与 App 登录使用等条件对照，只标注对进入对话有实际影响的端差异。
- Impact — 假设（Assumption）：预期证明跨端产品起点的一致性。缺口（Gap）：需原片确认。

### Web 02：启动后的对话互动

- Context — 事实（Fact）：`Designer-reported` Web 录屏展示启动后的 AI 对话互动。
- Problem — 假设（Assumption）：浏览器的屏幕空间、输入设备、摄像头 / 麦克风权限与窗口焦点不同于 App，但话轮反馈不能改变意义。
- Decision — 建议（Recommendation）：与 App 对话互动建立等价任务对比，区分必须共享的话轮状态与可以适配的布局、密度和控制方式。
- Impact — 假设（Assumption）：意图是降低跨端重新学习，并证明同一实时交互模型能跨端成立。缺口（Gap）：需原片、状态对齐和跨端使用证据。

## 7. 3 分钟叙事

> 事实（Fact）：根据我对项目的口述，ConvoAI 是一个面向客户体验演示的 1 对 1 AI 对话产品，同时覆盖 App 与 Web。它包含语音与视频对话、字幕、动感球、打断、硬件绑定和数字人等形态。我独立负责产品设计，产品已正式上线；这两项当前都是 `Designer-reported`，尚待项目记录与上线素材佐证。
>
> 假设（Assumption）：这个项目最值得讲的难题，不是如何做出一个类似 ChatGPT 的对话页，而是如何让用户理解一个不可见的实时系统。通话中，连接、聆听、AI 生成、音视频播放、字幕和数字人都在同时变化。只要它们之间的状态不一致，用户就会不知道 AI 是否听见、是否还在说，以及自己是否能打断。
>
> 建议（Recommendation）：案例因此围绕三个原则展开：让状态可感知，让用户可介入，让对话可恢复。第一个关键决策是进入通话时的就绪反馈；用户必须知道权限、连接和 AI 是否已经准备好。第二个决策是用球体、字幕、声音或数字人共同表达话轮，不让任何一个通道单独承担状态反馈。第三个决策是打断：它不只是停止播放，而是把话轮完整地交还给用户，并让 AI 重新进入聆听。
>
> 缺口（Gap）：当前还没有证据支持使用效率、满意度或商业影响。案例现阶段首先需要用已上线录屏证明设计如何把 RTC、RTM 与 AI 的复杂实时状态转化为人能理解的互动节奏；结果章仍只声称有独立证据支持的影响。
