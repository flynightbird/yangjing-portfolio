# ConvoAI 案例叙事蓝图

## 1. 项目定位

ConvoAI 是同时覆盖 App 与 Web 的 1 对 1 实时 AI 对话产品。与文字聊天不同，它要在同一次通话中连续处理连接、聆听、生成、播放、打断、字幕、视频与数字人。

案例的核心主题是：**让实时 AI 对话可感知、可介入、可恢复。**

设计职责覆盖产品结构、核心交互、跨端体验与实时状态的界面表达。

## 2. 叙事主线

整个案例围绕一个问题展开：当 AI 对话从单次输入变成持续通话，如何让用户理解当前状态，并始终保留对话的控制权？

叙事按照以下顺序推进：

1. 先解释实时对话为什么比文字聊天复杂。
2. 再建立完整的 App 产品结构。
3. 说明用户如何从进入产品走到第一个话轮。
4. 展开字幕、摄像头、打断与声纹中的控制权。
5. 将数字人作为更强临场感下的互动延伸。
6. 最后展开一次回答背后的实时处理链路。
7. 以跨端产品设计交付收束。

## 3. 章节架构

| 章节 | 核心问题 | 设计重点 |
|---|---|---|
| 背景与挑战 | 用户如何理解连续变化的对话状态？ | 用 Connecting 到 Recovery 建立状态语言 |
| App 产品结构 | 实时通话之外，产品如何承接完整任务？ | 组织 Agent、个人设置与硬件入口 |
| 开始实时对话 | 用户何时能够开始说话？ | 连续表达权限、连接、就绪与会话边界 |
| 对话控制 | 用户如何看懂状态并改变话轮？ | 稳定控制区、语音打断、声纹模式 |
| 数字人互动 | 更强的临场感下，如何保持话轮清晰？ | 连接角色选择、音画反馈与画中画 |
| 实时链路 | 一次回答经过了哪些处理阶段？ | 将 RTC、ASR、LLM、TTS 与 Voiceprint 放回对话语境 |
| 设计交付 | 跨端体验如何形成完整系统？ | 统一产品结构、状态语言与控制方式 |

## 4. 视觉方向

- 页面以近黑背景与灰阶层次为主，不用大面积亮色。
- 青色用于实时活动与当前状态，冷蓝用于系统处理，红色只用于打断与恢复。
- 巨大低透明度数字建立章节节奏，中文标题始终是第一阅读层级。
- 真实产品画面承担解释任务，不作为装饰。
- Hero 与实时链路是两个视觉峰值，其他章节保持稳定的阅读节奏。

## 5. 媒体编排

### App 产品结构

左侧依次说明登录与进入、产品页面结构、个人设置和绑定硬件设备；右侧手机在同一位置切换对应画面。

### 开始实时对话

按 App 启动对话、Web 登录、Web 启动前设置、Agent 选择布局、加入与退出会话的顺序，呈现从进入产品到开始说话的跨端路径。

### 对话控制

将字幕与摄像头、Web 连续对话、语音打断与声纹锁定放在同一章，说明信息反馈、话轮控制与响应边界如何配合。

### 数字人与实时链路

数字人选择与互动并排呈现。实时链路使用左侧对话画面与右侧处理阶段的 pinned 结构，随滚动依次点亮 RTC、ASR、LLM、TTS 与 Voiceprint。

## 6. 核心设计判断

| 场景 | 问题 | 设计选择 | 用户体验 |
|---|---|---|---|
| 权限与连接 | 等待容易被误解为无响应 | 将权限、Connecting 与 Ready 组织为连续过程 | 用户知道系统正在做什么，以及何时能够开口 |
| 连续对话 | 文本、球体、字幕、视频与控件同时变化 | 保持控制区稳定，让其他反馈围绕话轮更新 | 信息增加后仍然能追踪 AI 状态 |
| 语音打断 | 新意图需要同时改变声音、文本与状态 | 保留 Interrupted 内容，进入 Thinking 后恢复新回答 | 用户夺回话轮时不丢失上下文 |
| 声纹 | 便利性、身份控制与录入成本存在取舍 | 提供 Off、Seamless 与 Personalized 三种模式 | 用户可以选择 Agent 对说话人的响应边界 |
| 数字人 | 临场感放大音画同步与画中画的竞争 | 将连接、话轮、摄像头与画中画组织在同一流程 | 更强临场感下仍能看懂当前状态 |
| 实时数据 | 连续回答背后包含多个处理阶段 | 将各阶段数据放在正在发生的对话旁 | 对话体验与处理链路可以被同时理解 |

## 7. 3 分钟讲述

1. ConvoAI 是一个跨 App 与 Web 的 1 对 1 实时 AI 对话产品。
2. 核心挑战不是做一个聊天页，而是让用户在连续变化中看懂状态并保持控制。
3. 我从完整产品结构开始，统一梳理了进入产品、开始对话、连续交流、打断、声纹、数字人与实时数据。
4. App 与 Web 共享一致的状态语言，但根据各自的任务密度采用不同布局。
5. 最终产品通过可感知、可介入、可恢复的状态与控制语言，连接起完整的跨端对话体验。

## 9. 面试准备

| 可能问题 | 回答主线 |
|---|---|
| 你负责什么？ | 产品结构、核心交互、跨端体验与实时状态的界面表达 |
| 为什么需要七段状态？ | 它们把连接、聆听、生成、回答、打断与恢复组织成可理解的话轮 |
| App 和 Web 为什么不做成一样？ | App 优先快速进入对话，Web 承载更多启动前配置；两端统一的是状态语言，不是布局 |
| 语音打断如何保留上下文？ | 保留 Interrupted 内容，同步改变音频、文本、球体与状态，再进入新话轮 |
| 声纹模式如何取舍？ | Off、Seamless 与 Personalized 对应不同的便利性、身份控制与录入成本 |
| 数字人带来了什么新问题？ | 更强临场感会放大音画同步、画中画与话轮反馈的重要性 |
| 实时链路为什么要放在对话旁？ | 让用户体验与背后处理过程保持同一上下文 |

## 10. 案例质量检查

- 首屏是否在一个视口内说清产品、平台和核心挑战。
- 七章是否围绕用户的实时判断推进，而不是变成功能目录。
- App 与 Web 的共性与差异是否都有明确说明。
- 每个媒体是否都有清晰的场景、问题、设计与体验说明。
- 实时链路是否被清晰定位为对话过程的延展。
- 桌面、平板、手机与减弱动态模式下，阅读顺序与媒体控件是否仍然清晰。
