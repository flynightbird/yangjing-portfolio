**建议（Recommendation）：**

> ConvoAI 是面向客户体验演示的跨端 1 对 1 AI 对话产品；设计的核心不是展示更多 AI 能力，而是让不可见、随时变化的实时通话状态变得可感知、可介入、可恢复。

**作用域行：**独立负责产品设计（Designer-reported）；产品已正式上线（Designer-reported）。

## 4. 视觉方向

**建议（Recommendation）：**案例视觉语言表现“实时互动的节奏与状态”，不使用泛 AI 科幻感。

- 层级：首屏使用真实通话瞬间和一句核心转变。
- 字体：延续作品集的干净无衬线体；状态与时间使用等宽数字。
- 颜色：中性背景 + 一个产品主色；颜色仅编码 Listening / Thinking / Speaking / Interrupted / Recovery。
- 网格：桌面端使用 12 列；手机录屏保持原始宽高比和稳定尺寸。
- 密度：视频周围只保留一层 CPDI 说明，技术状态放在独立系统图。
- 节奏：长录屏不加速、不裁切；用封面帧、章节标题、完整时长标注与播放控件管理扫读节奏。

## 7. 3 分钟叙事

> ConvoAI 是一个面向客户体验演示的 1 对 1 AI 对话产品，同时覆盖 App 与 Web。我独立负责产品设计，产品已正式上线；这两项保留 `Designer-reported` 归属。
>
> 这个项目最值得讲的难题，不是如何做出一个类似 ChatGPT 的对话页，而是如何让用户理解一个不可见的实时系统。连接、聆听、AI 生成、音视频播放、字幕和数字人都在同时变化；它们一旦失去一致性，用户就会不知道 AI 是否听见、是否还在说，以及自己能否介入。
>
> 案例因此围绕三个原则展开：让状态可感知，让用户可介入，让对话可恢复。App 录屏展示从权限、连接到多模态对话，以及设置、硬件和数字人如何扩展通话体验。Web 录屏则完整展开另一层系统证据：用户如何选择 Agent、加入和退出，如何在连续聊天中语音打断 AI，以及如何用 RTC、ASR、LLM 与 TTS 延迟数据定位实时体验。
>
> 作品集会保留所有 App 原片的完整时长。编辑发生在视频之外：用章节分组、封面帧和 CPDI 说明帮助评审理解内容，不会以删减实时过程换取表面上的紧凑。

## 9. 面试问题

| 可能问题 | 回答要点 |
|---|---|
| 你具体负责了哪些？ | 独立负责产品设计；明确交互、信息结构、状态表达和跨端体验的边界 |
| 为什么不把它当成普通聊天产品？ | 实时对话中的连接、话轮、音视频与用户介入权会同时变化 |
| 为什么需要动感球？ | 球体承载连续的对话节奏与状态反馈，不只是装饰 |
| 打断在产品与技术上分别意味着什么？ | 用户意图、生成 / 音频 / 字幕停止、视觉状态转换与重新聆听 |

| 数字人带来什么特有问题？ | 音画同步、嘴型、凝视、字幕竞争、延迟可见性与画中画布局 |
| 为什么 App 和 Web 不能只使用响应式布局？ | 使用姿态、权限、摄像头、控制可达性与视觉焦点不同 |

| 最终什么上线了？ | 产品已正式上线（Designer-reported）；用 8 段 App 与 7 段 Web 录屏指明可见范围 |
| Web 端为什么要显示实时监控数据？ | 延迟感受可能来自 RTC、ASR、LLM、TTS 不同环节；分层数据让问题可定位、可演示、可与工程讨论 |
| 为什么录屏不裁切？ | 实时产品的价值在于状态连续性；删掉等待、转换或恢复会改变真实体验 |
| 项目产生了什么影响？ | 只说明已正式上线和已交付的跨端体验，不使用未经证实的转化或满意度数据 |
